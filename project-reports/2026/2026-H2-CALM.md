# Project Overview

CALM (the **Common Architecture Language Model**) is a declarative, JSON-based modelling language for describing complex software architectures — nodes, relationships, interfaces, controls and patterns — in a machine-readable, version-controlled form. It is designed for regulated environments such as financial services, where architecture must be validated, governed and audited as a first-class artefact rather than captured in static diagrams.

"Architecture as Code" (AasC) lets teams define an architecture once and then validate it against organisational standards, generate documentation and diagrams, and continuously check live systems for compliance with the intended design. CALM has been an open-source project from the outset, led by Morgan Stanley and developed in the open under FINOS; the specification has matured to **v1.2**. The project ships a full toolchain around the specification: a CLI, a backend "Hub" and UI, IDE tooling, a visual editor, a compliance platform, and AI-agent integration.

# Current Status

- **CALM specification advanced to v1.2**, introducing **Timelines** (capturing how an architecture evolves over time) and **Decorators** (cross-cutting metadata/overlays on model elements) alongside the existing node/relationship/interface/control schemas, pattern definitions and `$id`-driven addressing model — with support for both added across the tooling (Hub, CLI and MCP server).
- **CALM Hub** (Java/Quarkus backend) was reworked around the new `$id`-driven API structure, with standalone Nitrite and MongoDB storage modes and aligned deployment decorators.
- **CALM Hub UI / Visualizer 1.0** milestone delivered — the initial architecture visualiser reached completion.
- **CLI (`@finos/calm-cli`)** shipped continuously through to **v1.46.0**, including the `validate`, `generate`, `docify` and `template` commands and, most recently, a `--export-diagrams` flag that renders Mermaid diagrams via a local browser.
- **CALM Studio** — a new SvelteKit (Svelte 5) visual CALM editor — landed in the `calm-suite/` workspace.
- **CALMGuard** — a new Next.js continuous-compliance platform — landed alongside it, extending CALM from design-time modelling into runtime conformance checking.
- **VSCode extension**, **calm-widgets** (Handlebars-based visualisation/templating), and an **MCP server** for AI-agent integration were maintained and extended.
- **Documentation** at calm.finos.org expanded to cover `template`, `docify`, the widget library, and the CALM Studio tooling.
- **Ecosystem adoption** — [Archrails](https://archrails.io), a commercial architecture-governance platform, is built on FINOS CALM, enforcing CALM-defined rules at AI code-generation time (via MCP) and at pull-request time — an early example of CALM underpinning third-party products and validating its portable, vendor-agnostic standard.

# Challenges & Blockers

- **Contribution concentration / bus factor:** LFX flags that **2 contributors account for >51%** of contributions and **1 organisation accounts for >51%** of contributions, and this remains the project's primary health risk. The trend is, however, **improving**: measuring the top-2 human contributors' share of commits, concentration has fallen from **~61% over the trailing 365 days to ~53% over 180 days and ~49% over the last 90 days** (now below half), as more maintainers and organisations take on substantive work. Continuing to broaden the active maintainer and organisational base is the focus.
- **Adoption breadth:** strong tooling momentum, but production adoption signals (downloads, external case studies) need to grow beyond early adopters and the founding organisation.
- **Surface area vs. capacity:** the toolchain has expanded quickly (spec, Hub, UI, CLI, Studio, CALMGuard, VSCode, widgets, MCP), which raises maintenance and coordination load relative to the active contributor pool.
- **Specification stability:** as CALM 1.x stabilises, downstream tools must track schema changes without fragmenting the user experience.

# Roadmap & Goals for Next 6 Months

- **Consolidate the CALM Suite into the core monorepo** — bring CALM Studio and CALMGuard fully into the main `architecture-as-code` repository for a single build, release and governance surface, reducing fragmentation across the toolchain.
- **Interoperate with the FINOS AI Governance Framework (AIGF)** — use CALM to document AI patterns, modelling AI system architectures so they can be validated and governed against AIGF controls.
- **Stabilise the CALM 1.x specification** and the `$id`-driven Hub API, with clear versioning/compatibility guarantees for downstream tools.
- **CALM Studio** towards a GA-quality visual editing experience.
- **CALMGuard** maturation — continuous-compliance workflows from model to running system.
- **Hub & Visualizer** hardening (storage modes, deployment decorators, error surfacing).
- **Grow and diversify the contributor base** — onboarding docs, "good first issue" curation, and outreach to additional member organisations to reduce concentration.
- **Adoption** — reference implementations, tutorials (e.g. Advent of CALM), and conference/workshop presence to drive uptake.
- Assess readiness for a future **lifecycle transition** (Incubating → Graduated) against FINOS criteria.

# Additional Information

- CALM is an **npm-workspaces + Maven** monorepo: the specification (`calm/`), a TypeScript CLI, Java/Quarkus Hub backend, React Hub UI, shared models/widgets, a VSCode extension, the `calm-suite/` products (CALM Studio, CALMGuard), and AI-agent tooling all live together.
- Documentation: https://calm.finos.org · Advent of CALM educational series: https://calm.finos.org/advent/
- The project follows Conventional Commits, requires PR-based contribution with CI gates, and maintains a single root lockfile for reproducible cross-platform builds.
